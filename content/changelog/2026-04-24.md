## Spending limits

You can now set a **monthly spending limit** for your organization from the **Billing** page in the Neon Console. When spend reaches **80%** and **100%** of that limit, Neon sends email alerts to organization admins. Spend is evaluated about every **15 minutes**, and reminder emails continue weekly until you raise the limit or the billing cycle resets.

![Spending limit card on the Billing page](/docs/changelog/spending_limit.png)

When you enable a limit, you enter a dollar amount and choose threshold behavior. Email alerts at **80%** and **100%** are available now. In a future release, you will have the option to suspend computes automatically when the limit is reached.

![Enable spending limit dialog in Neon Console](/docs/changelog/set_spending_limit.png)

To get started, see [Spending limits](/docs/introduction/spending-limit).

## Monitor snapshot storage consumption

Building on the recent [snapshot size fields added to snapshot API responses](/docs/changelog/2026-04-17#snapshot-api-responses-include-storage-size-fields), the [Retrive project consumption metrics](https://api-docs.neon.tech/reference/getconsumptionhistoryperprojectv2) endpoint now supports a **`snapshot_storage_bytes_month`** in the `metrics` parameter. Use it to track monthly snapshot storage in your project-level consumption reporting.

Example response body (excerpt):

```json
{
  "timeframe_start": "2026-02-05T00:00:00Z",
  "timeframe_end": "2026-02-06T00:00:00Z",
  "metrics": [
    { "metric_name": "root_branch_bytes_month", "value": 758611968 },
    { "metric_name": "instant_restore_bytes_month", "value": 983488 },
    { "metric_name": "snapshot_storage_bytes_month", "value": 0 }
  ]
}
```

<Admonition type="note" title="Snapshot billing reminder">
Snapshot storage billing starts **May 1, 2026**. Until then, snapshots remain free during the beta period. See [Backup & restore](/docs/guides/backup-restore) and [Plans](/docs/introduction/plans) for pricing details.
</Admonition>

For the full metrics table, parameters, and examples, see [Querying consumption metrics](/docs/guides/consumption-metrics).

## Building durable AI agents with Pydantic AI, DBOS, and Neon

Multi-step agents depend on LLMs and tools that can time out, rate limit, or fail mid-run. A practical pattern is **durable execution**: checkpoint progress in **Postgres** so a workflow can resume after a crash or retry without redoing expensive steps from scratch. **DBOS** provides that durable execution layer, **Pydantic AI** structures agent logic and tool orchestration, and **Neon** backs the database so checkpoint state lives in serverless Postgres that scales with your workload.

For a full walkthrough with a working example, see [**Building Durable AI Agents with Pydantic AI, DBOS, and Neon**](/guides/pydantic-ai-dbos-neon).
